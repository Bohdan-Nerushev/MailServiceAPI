# План контейнеризації та розгортання застосунку

## 1. Підготовка інфраструктури (Віддалений сервер 172.18.6.19)
- Перевірка доступу до сервера по SSH: `ssh bnerushev@172.18.6.19`.
- Оновлення пакетної бази операційної системи (`sudo apt update && sudo apt upgrade -y`).
- Встановлення Docker Engine та плагіну Docker Compose (або валідація їх наявності та версій).
- Базове налаштування безпеки: конфігурація брандмауера (UFW) з відкриттям виключно необхідних портів (наприклад, 80, 443 для HTTP/HTTPS API; 25, 465, 587 для SMTP; 143, 993 для IMAP та 22 для SSH).

## 2. Контейнеризація основного застосунку
- Створення `Dockerfile` для API:
  - Використання офіційного, мінімального образу (наприклад, `python:3.11-slim`).
  - Копіювання файлу залежностей (`requirements.txt`) та встановлення пакетів без використання кешу (`pip install --no-cache-dir`).
  - Копіювання вихідного коду застосунку.
  - Налаштування запуску процесу від імені непривілейованого користувача для підвищення безпеки (не `root`).
  - Визначення `ENTRYPOINT` / `CMD` для запуску через production-ready HTTP сервер (наприклад, Gunicorn + Uvicorn).

## 3. Контейнеризація поштових серверів (Postfix та Dovecot)
- Систематизація існуючих налаштувань локального сервера: експорт `main.cf`, `master.cf` (для Postfix) та файлів директорії `/etc/dovecot/` у структуру проєкту.
- Підготовка `Dockerfile` для поштових сервісів. Для спрощення архітектури та керування можливо використання готових перевірених образів (наприклад, `tvial/docker-mailserver` або `analogj/docker-postfix`) або створення власних кастомних образів на базі Ubuntu.
- Визначення точок монтування (Volumes) для збереження стійкого стану (поштові скриньки, логіни).
- Підготовка TLS-сертифікатів для шифрування трафіку (SMTP/IMAP) та визначення безпечного механізму їх монтування всередину контейнерів.

## 4. Оркестрація компонентів через Docker Compose
- Створення `docker-compose.yml` у корені проєкту.
- Визначення сервісів: `api` (основний застосунок), `postfix` (SMTP сервер), `dovecot` (IMAP сервер).
- Налаштування ізольованої внутрішньої Docker-мережі для комунікації між API та поштовими контейнерами без експозиції зайвих портів на хост.
- Налаштування політик перезапуску компонентів: `restart: unless-stopped`.

## 5. Доставка (Deployment) на віддалений сервер
- **Локальний етап**: збірка образів (`docker-compose build`).
- **Транспортний етап (без використання Registry)**: 
  - Експорт Docker-образу основного застосунку у tar-архів: `docker save -o api_image.tar <image_name>`.
  - Передача архіву, конфігураційних файлів поштових серверів та `docker-compose.yml` на цільовий сервер через SCP: `scp api_image.tar docker-compose.yml config/ bnerushev@172.18.6.19:/шлях/до/робочої/директорії/`.
- **Етап розгортання (Віддалений сервер)**:
  - Імпорт образу в локальне сховище Docker: `docker load -i api_image.tar`.
  - Запуск застосунку у фоновому режимі: `docker compose up -d`.

## 6. Валідація та перевірка працездатності
- Перевірка стану запущених контейнерів та відсутності перезапусків (`docker compose ps`).
- Огляд системних логів застосунку та поштових сервісів (`docker compose logs -f`).
- Перевірка доступності Health Check ендпойнтів застосунку.
- Тестування кінцевого функціоналу: авторизація, відправлення (SMTP) та отримання (IMAP) пошти з віддаленого сервера.
